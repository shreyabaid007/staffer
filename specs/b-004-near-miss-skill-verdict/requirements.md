# Requirements — b-004 Near-Miss Skill Verdict + Selection Rationale + Closest-on-Skills

> Slice B-4. Enriches the **no-match** output in three ways (Part 3 folded in post-AD-097; b-004 is
> unmerged, so it ships as one slice/PR):
>
> **Part 1 (deterministic) — hard-skill verdict.** Today a no-match labels an availability
> near-miss "available 1 day after deadline" — but that candidate failed at the **availability
> gate**, *before* the hard-skill filter ever ran on them (`run_match`: gate → exact-skill-filter
> are two sequential narrowing stages; a gate failure never reaches the skill filter). So the
> label silently hides whether shifting the start date would actually make them eligible: they may
> still be dropped by `exact_hard_skill_filter`. Part 1 (per **AD-097**, which superseded the
> originally-signed AD-095 labelling approach) makes a near-miss require **clearing the hard
> skills**: only availability/location misses that pass `exact_hard_skill_filter` are surfaced;
> skill-failers are dropped from near-misses (still in the `exclusion_log`). Deterministic,
> LLM-free, no model change.
>
> **Part 2 (LLM) — selection rationale.** Adds a new `NearMiss.selection_rationale` field: a short
> prose "why this candidate is worth considering once the gap is resolved", generated by an LLM
> through `PseudonymisedLM` for the **displayed (top-3) near-misses only**. This part **does**
> touch the frozen `dsm/models.py` contract (an additive optional field, AD-060 amendment) and
> introduces an injected LLM seam threaded through `run_match` → `_no_match` → near-miss assembly —
> symmetric to how the shortlist's `CandidateAssessment.narrative` is produced. Chosen over a
> deterministic strengths summary at the design fork (user decision).
>
> **Part 3 — closest-on-skills (AD-098).** Because AD-097 (Part 1) drops skill-failers from
> near-misses, a no-match where everyone available is a skill short (e.g. ROLE-05) shows *zero*
> near-misses. Part 3 adds a **separate, disjoint** `NoMatchResult.closest_on_skills` list:
> candidates who cleared **both** gates and failed **only** the hard-skill filter, ranked by fewest
> hard skills missing, capped at 3, reusing the Part-2 rationale. It is the mirror of a near-miss
> (near-miss = clears skills, one gate away; closest-on-skills = clears gates, a skill short) and
> never pollutes the actionable near-miss list. Additive frozen-contract field (AD-060). See **§
> Part 3** in `design.md`.
>
> Source: `dsm/cli/commands.py::build_near_misses` + the no-match path in `run_match`. Pipeline
> order: gate `filter_candidates` → `exact_hard_skill_filter`. Settled semantics this slice
> refines: **AD-063** (near-miss assembly), **AD-088** (hard-skill miss ordering).

## User story

As a staffer reading a no-match result, when I see a near-miss like "available 1 day after the
deadline", I want to know whether that candidate would *actually* qualify if I shifted the start
date — or whether they would still be rejected on skills — so I don't act on a half-truth and
move a date for someone who can't do the job anyway.

## Problem statement (current behaviour)

- `build_near_misses` turns each `Exclusion` into a `NearMiss` by recomputing a gap from the
  structured `Candidate` + `TargetProfileScorecard` it holds (AD-063c — it must **not** parse
  `Exclusion.detail`).
- An `AVAILABILITY_MISMATCH` near-miss reports only the day overshoot. A `LOCATION_MISMATCH`
  near-miss reports only the city gap. **Both candidates were excluded by a gate, so their hard
  skills were never evaluated** — the skill filter runs only on gate survivors
  (`run_match` lines ~191–207).
- Consequence: "available 1 day after deadline" implies "shift the date and they're in", which
  is false whenever the candidate also lacks a hard skill. Only `HARD_SKILL_MISMATCH` near-misses
  (which by construction already cleared location + availability) carry a complete picture.

## Product invariants referenced

- **AD-002 / Golden rule 2** — eligibility is deterministic + LLM-free. The skill verdict (Part 1)
  SHALL be computed by the existing deterministic filter, never an LLM. The selection rationale
  (Part 2) is **explanation, not eligibility** — it never changes who is or isn't a near-miss, so
  it may use an LLM without violating this rule (parallel to the shortlist narrative).
- **Golden rule 3 / no bare LLM calls / no-PII-leak** — every LLM call goes through
  `PseudonymisedLM`; no `name`/`email` reaches a provider. The selection-rationale predictor SHALL
  be built over `PseudonymisedLM` exactly like `make_score_predictor` / `make_clarify_predictor`.
- **AD-063b/c/d** — ordered, capped (top-3) near-misses; gaps recomputed from structured data,
  `Exclusion.detail` never parsed. Carried; this slice extends the gap content and refines
  intra-type ordering only.
- **AD-088** — hard-skill misses rank below availability misses; among them, fewest missing
  skills first. Carried; this slice adds an availability sub-ordering (see FR-6).
- **AD-072 / AD-033** — exact hard-skill membership, no adjacency credit, proficiency floor
  enforced. The verdict SHALL reuse this exact logic, not a looser re-implementation.
- **AD-060** — `dsm/models.py` is frozen. This slice introduces **no** model change (see NF-3).

## ADRs to ratify at the `T-000-ADR` gate (used: AD-095, AD-096, AD-097, AD-098)

- **AD-095 · Near-misses report their hard-skill verdict** *(superseded by AD-097)* — Originally:
  every pre-skill-filter near-miss labels its hard-skill verdict in `gap_summary` (kept the
  skill-failers, annotated them). Superseded — see AD-097.

- **AD-097 · A near-miss must clear the hard skills** — A near-miss is a candidate **one fixable
  decision** from eligibility: it fails only a negotiable gate (availability/location) **and**
  clears the non-negotiable hard skills. `build_near_misses` keeps only availability/location
  misses that clear `exact_hard_skill_filter`; pure `HARD_SKILL_MISMATCH` misses **and** gate
  misses that also fail a hard skill are dropped from near-misses (still recorded in the
  `exclusion_log` — the transparency layer). **Supersedes AD-088** (hard-skill misses as
  near-misses) and **AD-095** (label-the-gap). `gap_summary` reverts to plain negotiable-gap
  wording; no skill suffix, no sub-ordering. No frozen-model change.

- **AD-096 · `NearMiss.selection_rationale` (LLM, top-3 only, via `PseudonymisedLM`)** — Add an
  optional `selection_rationale: str | None = None` field to `NearMiss` carrying a short LLM-written
  "why consider this candidate once the gap is resolved". **Frozen-contract amendment (AD-060)** —
  additive + optional + defaulted, so all existing `NearMiss(...)` constructions keep validating.
  Generated **only for the ≤3 near-misses actually shown** (after ordering + cap), through an
  injected predictor seam over `PseudonymisedLM` (no PII to providers, Golden rule 3), mirroring
  `score_candidate`'s narrative. On predictor error the field stays `None` (the near-miss is still
  shown) — parallel to `score_candidate`'s skip-on-error. The rationale is **explanation only**: it
  never changes the near-miss set or ordering, so the determinism invariant continues to bind the
  set + order, not the prose (same scoping as the shortlist narrative). Unaffected by AD-097 — it
  annotates whatever near-misses remain. Rejected alternative: deterministic structured strengths
  summary (LLM-free) — simpler and bit-deterministic, but the user chose richer LLM prose.

- **AD-098 · `NoMatchResult.closest_on_skills` (Part 3)** — Add an optional
  `closest_on_skills: list[NearMiss] = Field(default_factory=list)` to the frozen `NoMatchResult`,
  carrying candidates who cleared **both** gates and failed only the hard-skill filter
  (`HARD_SKILL_MISMATCH`), ranked by fewest hard skills missing, capped at 3, each optionally
  annotated with the AD-096 `selection_rationale`. **Frozen-contract amendment (AD-060)** — additive,
  optional, defaulted. The structural counterpart to AD-097: together near-misses + closest-on-skills
  cover "one axis away" (negotiable gate / skill axis). Reuses the `NearMiss` type. Rejected:
  re-admitting skill-failers to `near_misses` — exactly what AD-097 removed; a separate field keeps
  the actionable list clean.

- **AD-096 · `NearMiss.selection_rationale` (LLM, top-3 only, via `PseudonymisedLM`)** — Add an
  optional `selection_rationale: str | None = None` field to `NearMiss` carrying a short LLM-written
  "why consider this candidate once the gap is resolved". **Frozen-contract amendment (AD-060)** —
  additive + optional + defaulted, so all existing `NearMiss(...)` constructions keep validating.
  Generated **only for the ≤3 near-misses actually shown** (after ordering + cap), through an
  injected predictor seam over `PseudonymisedLM` (no PII to providers, Golden rule 3), mirroring
  `score_candidate`'s narrative. On predictor error the field stays `None` (the near-miss is still
  shown) — parallel to `score_candidate`'s skip-on-error. The rationale is **explanation only**: it
  never changes the near-miss set or ordering, so the determinism invariant continues to bind the
  set + order, not the prose (same scoping as the shortlist narrative). Rejected alternative:
  deterministic structured strengths summary (LLM-free) — simpler and bit-deterministic, but the
  user chose richer LLM prose at the design fork.

## Functional requirements (EARS)

> **Part 1 reframed by AD-097 (post-sign-off).** The originally-signed Part 1 *labelled* the skill
> gap in `gap_summary` (AD-095). On review (the "why surface a candidate a date shift won't fix?"
> question), it was changed to **exclude** skill-failers from near-misses entirely (AD-097
> supersedes AD-095). FR-1/FR-2/FR-5/FR-6/FR-7 below are restated to the AD-097 behaviour.

### FR-1 — Availability near-miss requires clearing hard skills

- **FR-1-AC-1** — WHEN `build_near_misses` processes an `AVAILABILITY_MISMATCH` exclusion, the
  system SHALL evaluate that candidate against `scorecard.hard_depth_skills` and include them as a
  near-miss **only if they clear every hard skill**.
- **FR-1-AC-2** — WHEN the candidate clears, the near-miss `gap_summary` SHALL be the plain
  negotiable-gap wording (`"available N day(s) after deadline"`) — no skill suffix, since every
  surfaced near-miss clears skills by construction.
- **FR-1-AC-3** — WHEN the candidate misses ≥1 hard skill (or holds it below the proficiency
  floor), they SHALL NOT appear as a near-miss; they remain recorded in the `exclusion_log`.

### FR-2 — Location near-miss requires clearing hard skills (consistency)

- **FR-2-AC-1** — WHEN `build_near_misses` processes a `LOCATION_MISMATCH` exclusion, the system
  SHALL include it as a near-miss only if the candidate clears every hard skill; the `gap_summary`
  is the plain city-gap wording (`"in {city}, not in onsite set for {role_city}"`).

### FR-3 — Verdict reuses the deterministic filter (no duplication)

- **FR-3-AC-1** — The clear/not-clear verdict SHALL be computed by reusing
  `dsm/index/retrieve.py::exact_hard_skill_filter` (already imported in `commands.py`), so it can
  never diverge from the real gate: same membership rule, same proficiency floor (AD-072), no
  adjacency credit (AD-033).
- **FR-3-AC-2** — The verdict SHALL be deterministic and LLM-free (Golden rule 2 / AD-002).

### FR-4 — `Exclusion.detail` still never parsed (AD-063c preserved)

- **FR-4-AC-1** — The verdict SHALL be derived from the structured `Candidate` +
  `TargetProfileScorecard` objects only. The system SHALL NOT read or parse `Exclusion.detail`.

### FR-5 — Empty hard-skill requirement

- **FR-5-AC-1** — WHEN `scorecard.hard_depth_skills` is empty (no hard requirement), every
  negotiable-gate miss clears (the filter passes everyone) and SHALL appear as a near-miss,
  consistent with `exact_hard_skill_filter`'s empty-list behaviour.

### FR-6 — Ordering (AD-063b, unchanged)

- **FR-6-AC-1** — Near-misses SHALL be ordered: **availability** misses first (smallest
  day-overshoot first), then **location** misses; each tie-broken by `candidate_email`. (No
  skill-based sub-ordering — all near-misses clear skills, so the AD-095 sub-rank is dropped.)
- **FR-6-AC-2** — The top-3 cap (AD-063d) is unchanged.

### FR-7 — Hard-skill failures are not near-misses (supersedes AD-088)

- **FR-7-AC-1** — A `HARD_SKILL_MISMATCH` exclusion SHALL NOT produce a near-miss. Such candidates
  (who cleared location + availability but lack a hard skill) are recorded in the `exclusion_log`
  only — a hard-skill gap has no negotiable fix, so it is not "near".

### FR-8 — Part 1 confined to near-miss assembly

- **FR-8-AC-1** — The Part-1 (skill-verdict) change SHALL be confined to
  `dsm/cli/commands.py::build_near_misses`. The gates (`filter_candidates`), the exact filter,
  scoring, and ranking SHALL be byte-for-byte unchanged. (Part 2 additionally wires an LLM seam
  through the no-match path — see FR-9/FR-10.)

### FR-9 — Selection rationale field (Part 2, LLM)

- **FR-9-AC-1** — `dsm/models.py::NearMiss` SHALL gain an optional `selection_rationale: str | None
  = None` field. The default keeps every existing construction valid (FR backward-compat).
- **FR-9-AC-2** — WHEN a no-match result is assembled, the system SHALL generate a
  `selection_rationale` for **each of the ≤3 near-misses that will be returned** (after ordering +
  the AD-063d cap), and SHALL NOT generate one for near-misses beyond the cap.
- **FR-9-AC-3** — The rationale SHALL describe the candidate's positive fit signals (e.g.
  hard/desired-skill coverage, experience, positive/retention feedback) framed as "why consider
  them once the {gap} is resolved" — complementing, not repeating, `gap_summary`.
- **FR-9-AC-4** — WHEN the rationale predictor errors or is absent, `selection_rationale` SHALL be
  `None` and the near-miss SHALL still be returned (parallel to `score_candidate` skip-on-error).

### FR-10 — Rationale goes through the PII boundary (Golden rule 3)

- **FR-10-AC-1** — The rationale predictor SHALL be built over `PseudonymisedLM` (like
  `make_score_predictor`); no `name`/`email` SHALL reach OpenRouter or Modal. (In current serving,
  `Candidate.email`/`name` already carry the pseudonymised `candidate_id` per AD-091; the
  `PseudonymisedLM` routing is mandatory regardless, for correctness and future hydration.)
- **FR-10-AC-2** — The predictor SHALL be an **injected seam** (a `Callable` type alias, like
  `ScorePredictor`), threaded `run_match` → `_no_match` → assembly, so unit tests inject a fake and
  run with no network (NF-1). The live predictor is built only at the CLI edge.

### FR-11 — Closest-on-skills population (Part 3, AD-098)

- **FR-11-AC-1** — `NoMatchResult.closest_on_skills` SHALL contain exactly the candidates whose
  exclusion reason is `HARD_SKILL_MISMATCH` (which, by pipeline order, cleared location +
  availability and failed only the exact hard-skill filter).
- **FR-11-AC-2** — A candidate that failed a gate SHALL NOT appear here (double-misses stay in the
  `exclusion_log` only); `near_misses` and `closest_on_skills` SHALL be disjoint.

### FR-12 — Closest-on-skills ordering, gap, rationale

- **FR-12-AC-1** — Entries SHALL be ordered by **fewest hard-skill gaps first** (missing +
  below-floor count ascending), then `candidate_email`; capped at 3 (AD-063d).
- **FR-12-AC-2** — `gap_summary` SHALL state the hard-skill gap (e.g. `"missing 1 hard skill:
  spring boot"`, incl. below-floor shortfalls), derived from structured data — `Exclusion.detail`
  SHALL NOT be parsed (AD-063c) — and computed by the **same** logic as `exact_hard_skill_filter`
  (one source of truth, AD-072/033), via a shared helper.
- **FR-12-AC-3** — Each of the ≤3 shown entries SHALL get a `selection_rationale` via the existing
  AD-096 seam (`PseudonymisedLM`, capped, error → `None`).

### FR-13 — Closest-on-skills surfaced + bounded

- **FR-13-AC-1** — `closest_on_skills` SHALL serialise in the `NoMatchResult` JSON and appear in the
  `dsm explain` no-match lineage.
- **FR-13-AC-2** — WHEN the no-match is gate-only or empty-input (no `HARD_SKILL_MISMATCH`
  exclusions), `closest_on_skills` SHALL be empty. `build_near_misses` (AD-097) SHALL be unchanged.

## Non-functional requirements

- **NF-1** — No network / no real LLM in tests; the rationale seam is injected as a fake predictor
  and the skill-verdict path is pure Python (mirrors `tests/cli/test_no_match.py`).
- **NF-2** — No new dependency (DSPy / `PseudonymisedLM` already in `docs/tech.md` + in use).
- **NF-3** — Part 1 adds no model change (AD-097 filters near-misses; `gap_summary` keeps its
  existing plain wording). Part 2's **only** `dsm/models.py` change is the additive, optional,
  defaulted `NearMiss.selection_rationale` (AD-096, an AD-060 amendment); all other models
  unchanged.
- **NF-4** — Gates stay deterministic + LLM-free (Golden rule 2). The LLM seam touches only the
  near-miss explanation, never eligibility or ordering.
- **NF-5** — `make check` green.

## Out of scope

- Re-checking **location** eligibility for availability misses or vice-versa (location is
  structural; AD-097 only adds the hard-skill clearance requirement).
- Any change to gate order, the exact filter, scoring, ranking, or the `explain` lineage shape
  beyond the now-filtered near-miss list and the new `selection_rationale`.
- Surfacing the rationale on the **shortlist** (`ShortlistResult`) path — near-miss only.
- Generating a rationale for near-misses beyond the top-3 cap (cost control, FR-9-AC-2).
- A deterministic strengths summary (the rejected fork alternative, AD-096).
