# Requirements — b-004 Near-Miss Skill Verdict + Selection Rationale

> Slice B-4. Enriches the **no-match near-miss** output in two ways:
>
> **Part 1 (deterministic) — hard-skill verdict.** Today a no-match labels an availability
> near-miss "available 1 day after deadline" — but that candidate failed at the **availability
> gate**, *before* the hard-skill filter ever ran on them (`run_match`: gate → exact-skill-filter
> are two sequential narrowing stages; a gate failure never reaches the skill filter). So the
> label silently hides whether shifting the start date would actually make them eligible: they may
> still be dropped by `exact_hard_skill_filter`. Part 1 makes every pre-skill-filter near-miss
> **report its hard-skill verdict** in `gap_summary`, so the label tells the whole truth. This part
> stays deterministic, LLM-free, and inside the existing `gap_summary` string — no model change.
>
> **Part 2 (LLM) — selection rationale.** Adds a new `NearMiss.selection_rationale` field: a short
> prose "why this candidate is worth considering once the gap is resolved", generated by an LLM
> through `PseudonymisedLM` for the **displayed (top-3) near-misses only**. This part **does**
> touch the frozen `dsm/models.py` contract (an additive optional field, AD-060 amendment) and
> introduces an injected LLM seam threaded through `run_match` → `_no_match` → near-miss assembly —
> symmetric to how the shortlist's `CandidateAssessment.narrative` is produced. Chosen over a
> deterministic strengths summary at the design fork (user decision).
>
> Source: `dsm/cli/commands.py::build_near_misses` + the no-match path in `run_match`. Pipeline
> order: gate `filter_candidates` → `exact_hard_skill_filter`. Settled semantics this slice
> refines: **AD-063** (near-miss assembly), **AD-088** (hard-skill miss ordering).

## User story

As a staffer reading a no-match result, when I see a near-miss like "available 1 day after the
deadline", I want to know whether that candidate would *actually* qualify if I shifted the start
date — or whether they would still be rejected on skills — so I don't act on a half-truth and
move a date for someone who can't do the job anyway.

## Problem statement (current behaviour)

- `build_near_misses` turns each `Exclusion` into a `NearMiss` by recomputing a gap from the
  structured `Candidate` + `TargetProfileScorecard` it holds (AD-063c — it must **not** parse
  `Exclusion.detail`).
- An `AVAILABILITY_MISMATCH` near-miss reports only the day overshoot. A `LOCATION_MISMATCH`
  near-miss reports only the city gap. **Both candidates were excluded by a gate, so their hard
  skills were never evaluated** — the skill filter runs only on gate survivors
  (`run_match` lines ~191–207).
- Consequence: "available 1 day after deadline" implies "shift the date and they're in", which
  is false whenever the candidate also lacks a hard skill. Only `HARD_SKILL_MISMATCH` near-misses
  (which by construction already cleared location + availability) carry a complete picture.

## Product invariants referenced

- **AD-002 / Golden rule 2** — eligibility is deterministic + LLM-free. The skill verdict (Part 1)
  SHALL be computed by the existing deterministic filter, never an LLM. The selection rationale
  (Part 2) is **explanation, not eligibility** — it never changes who is or isn't a near-miss, so
  it may use an LLM without violating this rule (parallel to the shortlist narrative).
- **Golden rule 3 / no bare LLM calls / no-PII-leak** — every LLM call goes through
  `PseudonymisedLM`; no `name`/`email` reaches a provider. The selection-rationale predictor SHALL
  be built over `PseudonymisedLM` exactly like `make_score_predictor` / `make_clarify_predictor`.
- **AD-063b/c/d** — ordered, capped (top-3) near-misses; gaps recomputed from structured data,
  `Exclusion.detail` never parsed. Carried; this slice extends the gap content and refines
  intra-type ordering only.
- **AD-088** — hard-skill misses rank below availability misses; among them, fewest missing
  skills first. Carried; this slice adds an availability sub-ordering (see FR-6).
- **AD-072 / AD-033** — exact hard-skill membership, no adjacency credit, proficiency floor
  enforced. The verdict SHALL reuse this exact logic, not a looser re-implementation.
- **AD-060** — `dsm/models.py` is frozen. This slice introduces **no** model change (see NF-3).

## ADRs to ratify at the `T-000-ADR` gate (next free IDs: AD-095, AD-096)

- **AD-095 · Near-misses report their hard-skill verdict** — Every near-miss whose candidate was
  excluded *before* the hard-skill filter (`AVAILABILITY_MISMATCH`, `LOCATION_MISMATCH`) SHALL
  have its `gap_summary` state whether the candidate clears the role's hard skills, computed by
  reusing `exact_hard_skill_filter` (deterministic, no adjacency, floor-enforced). Availability
  near-misses that **also** miss a hard skill rank below those that clear skills (the latter are
  the genuinely actionable ones). **No `NearMiss` field is added** — the verdict rides in the
  existing `gap_summary` string, so AD-060 is untouched. Refines AD-063b ordering and the AD-088
  rationale (actionability).

- **AD-096 · `NearMiss.selection_rationale` (LLM, top-3 only, via `PseudonymisedLM`)** — Add an
  optional `selection_rationale: str | None = None` field to `NearMiss` carrying a short LLM-written
  "why consider this candidate once the gap is resolved". **Frozen-contract amendment (AD-060)** —
  additive + optional + defaulted, so all existing `NearMiss(...)` constructions keep validating.
  Generated **only for the ≤3 near-misses actually shown** (after ordering + cap), through an
  injected predictor seam over `PseudonymisedLM` (no PII to providers, Golden rule 3), mirroring
  `score_candidate`'s narrative. On predictor error the field stays `None` (the near-miss is still
  shown) — parallel to `score_candidate`'s skip-on-error. The rationale is **explanation only**: it
  never changes the near-miss set or ordering, so the determinism invariant continues to bind the
  set + order, not the prose (same scoping as the shortlist narrative). Rejected alternative:
  deterministic structured strengths summary (LLM-free) — simpler and bit-deterministic, but the
  user chose richer LLM prose at the design fork.

## Functional requirements (EARS)

### FR-1 — Availability near-miss reports skill verdict

- **FR-1-AC-1** — WHEN `build_near_misses` builds a near-miss for an `AVAILABILITY_MISMATCH`
  exclusion, the system SHALL evaluate that candidate against `scorecard.hard_depth_skills` and
  include the verdict in `gap_summary`.
- **FR-1-AC-2** — WHEN the candidate clears every hard skill, the `gap_summary` SHALL state that
  they clear hard skills (e.g. `"available 1 day after deadline; clears all hard skills"`), so a
  reader knows a date shift alone makes them eligible.
- **FR-1-AC-3** — WHEN the candidate misses ≥1 hard skill, the `gap_summary` SHALL state that a
  skill gap remains (e.g. `"available 1 day after deadline; also missing 2 hard skills: kafka,
  spark"`), so a reader knows a date shift alone is **not** sufficient.

### FR-2 — Location near-miss reports skill verdict (consistency)

- **FR-2-AC-1** — WHEN `build_near_misses` builds a near-miss for a `LOCATION_MISMATCH`
  exclusion (also excluded before the skill filter), the system SHALL include the same hard-skill
  verdict in its `gap_summary`, appended to the existing city-gap wording.

### FR-3 — Verdict reuses the deterministic filter (no duplication)

- **FR-3-AC-1** — The skill verdict SHALL be computed by reusing
  `dsm/index/retrieve.py::exact_hard_skill_filter` (already imported in `commands.py`), so the
  near-miss verdict can never diverge from the real gate: same membership rule, same proficiency
  floor (AD-072), no adjacency credit (AD-033).
- **FR-3-AC-2** — The verdict SHALL be deterministic and LLM-free (Golden rule 2 / AD-002).

### FR-4 — `Exclusion.detail` still never parsed (AD-063c preserved)

- **FR-4-AC-1** — The verdict and the missing-skill names SHALL be derived from the structured
  `Candidate` + `TargetProfileScorecard` objects only. The system SHALL NOT read or parse
  `Exclusion.detail`.

### FR-5 — Empty hard-skill requirement

- **FR-5-AC-1** — WHEN `scorecard.hard_depth_skills` is empty (no hard requirement), every
  pre-skill-filter near-miss SHALL report that hard skills are cleared (the filter passes
  everyone), consistent with `exact_hard_skill_filter`'s empty-list behaviour.

### FR-6 — Ordering: actionable availability misses first (refines AD-063b)

- **FR-6-AC-1** — Among `AVAILABILITY_MISMATCH` near-misses, those that **clear** all hard skills
  SHALL rank above those that still miss a hard skill (the former are the only truly actionable
  ones — shift the date and they qualify). Within each group, the existing order holds: smallest
  day-overshoot first, then `candidate_email`.
- **FR-6-AC-2** — Cross-type ordering SHALL be unchanged: all availability misses rank above
  location misses (AD-063b), which rank above hard-skill misses (AD-088). The top-3 cap (AD-063d)
  is unchanged.

### FR-7 — Hard-skill near-misses unchanged

- **FR-7-AC-1** — `HARD_SKILL_MISMATCH` near-misses (whose candidates already cleared location +
  availability) SHALL keep their current `gap_summary` and ordering. No skill re-verdict is added
  — their gap is already the skill gap.

### FR-8 — Part 1 confined to near-miss assembly

- **FR-8-AC-1** — The Part-1 (skill-verdict) change SHALL be confined to
  `dsm/cli/commands.py::build_near_misses`. The gates (`filter_candidates`), the exact filter,
  scoring, and ranking SHALL be byte-for-byte unchanged. (Part 2 additionally wires an LLM seam
  through the no-match path — see FR-9/FR-10.)

### FR-9 — Selection rationale field (Part 2, LLM)

- **FR-9-AC-1** — `dsm/models.py::NearMiss` SHALL gain an optional `selection_rationale: str | None
  = None` field. The default keeps every existing construction valid (FR backward-compat).
- **FR-9-AC-2** — WHEN a no-match result is assembled, the system SHALL generate a
  `selection_rationale` for **each of the ≤3 near-misses that will be returned** (after ordering +
  the AD-063d cap), and SHALL NOT generate one for near-misses beyond the cap.
- **FR-9-AC-3** — The rationale SHALL describe the candidate's positive fit signals (e.g.
  hard/desired-skill coverage, experience, positive/retention feedback) framed as "why consider
  them once the {gap} is resolved" — complementing, not repeating, `gap_summary`.
- **FR-9-AC-4** — WHEN the rationale predictor errors or is absent, `selection_rationale` SHALL be
  `None` and the near-miss SHALL still be returned (parallel to `score_candidate` skip-on-error).

### FR-10 — Rationale goes through the PII boundary (Golden rule 3)

- **FR-10-AC-1** — The rationale predictor SHALL be built over `PseudonymisedLM` (like
  `make_score_predictor`); no `name`/`email` SHALL reach OpenRouter or Modal. (In current serving,
  `Candidate.email`/`name` already carry the pseudonymised `candidate_id` per AD-091; the
  `PseudonymisedLM` routing is mandatory regardless, for correctness and future hydration.)
- **FR-10-AC-2** — The predictor SHALL be an **injected seam** (a `Callable` type alias, like
  `ScorePredictor`), threaded `run_match` → `_no_match` → assembly, so unit tests inject a fake and
  run with no network (NF-1). The live predictor is built only at the CLI edge.

## Non-functional requirements

- **NF-1** — No network / no real LLM in tests; the rationale seam is injected as a fake predictor
  and the skill-verdict path is pure Python (mirrors `tests/cli/test_no_match.py`).
- **NF-2** — No new dependency (DSPy / `PseudonymisedLM` already in `docs/tech.md` + in use).
- **NF-3** — Part 1 adds no model change (verdict rides in `gap_summary`). Part 2's **only**
  `dsm/models.py` change is the additive, optional, defaulted `NearMiss.selection_rationale`
  (AD-096, an AD-060 amendment); `NoMatchResult` and all other models unchanged.
- **NF-4** — Gates stay deterministic + LLM-free (Golden rule 2). The LLM seam touches only the
  near-miss explanation, never eligibility or ordering.
- **NF-5** — `make check` green.

## Out of scope

- Re-checking **location** eligibility for availability misses or vice-versa (location is
  structural; only the skill verdict + rationale are added).
- Any change to gate order, the exact filter, scoring, ranking, or the `explain` lineage shape
  beyond the enriched `gap_summary` and the new `selection_rationale` already carried through.
- Surfacing the verdict/rationale on the **shortlist** (`ShortlistResult`) path — near-miss only.
- Generating a rationale for near-misses beyond the top-3 cap (cost control, FR-9-AC-2).
- A deterministic strengths summary (the rejected fork alternative, AD-096).
