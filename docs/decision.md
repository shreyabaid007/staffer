# Decision Log (ADRs) — Demand–Supply Matcher

> The repository's memory. **Read before proposing alternatives.** To change a decision, add a new entry that supersedes the old — never silently diverge.
> Format: **ID · Title** — *Status* — Decision — why / consequence.

## Architecture & method

- **AD-001 · Structured RAG, not agentic** — Accepted — LLM bounded to typed clarify + score steps; retrieval deterministic. Why: explainability, reproducibility, tractable evals, predictable cost.
- **AD-002 · Deterministic, LLM-free gates** — Accepted — Location + availability filtering is pure Python; an LLM cannot override eligibility. Why: trust boundary + fairness floor; fully unit-testable.
- **AD-003 · Spec-driven + steering files** — Accepted — Steering docs + per-feature specs are the source of truth; code derives from specs. Why: prevent drift, preserve context.

## PII & data

- **AD-010 · PII pseudonymisation is mandatory** — Accepted — All OpenRouter calls go through `PseudonymisedLM`; mapping in-memory only. Why: resumes are PII-dense; external LLM exposure is high-risk. — *Refined by AD-068 (ingestion-time identity is persisted in an encrypted vault, not in-memory) and AD-069 (deterministic redact-first + outbound leak-scan); the live-call in-memory mapping for OpenRouter is unchanged.*
- **AD-011 · Embedder receives PII-free text** — Accepted — `name`/`email` excluded from embedding input by construction; Modal sees no identity. Why: keeps the boundary intact while hosting the embedder externally.
- **AD-012 · Email is the join key** — Accepted — Join supply ↔ profiles ↔ feedback on email; first names collide in the data. — *Refined by AD-065 (supply sources are CSV snapshots, not xlsx) and AD-067 (email is tokenized to `candidate_id` = HMAC(email), which is the stable internal key downstream; name is never a join key).*
- **AD-013 · Candidate universe = supply sheets** — Accepted — Candidates are the people in Beach / Rolling Off / New Joiners; profiles enrich. A profile with no supply row = staffed → not a candidate.

## Gating & near-miss semantics

> **⚠ Provisional.** AD-063 was accepted alongside c-001. Gating semantics and near-miss assembly will be revisited as part of the query-time pipeline architecture. See AD-085.

- **AD-063 · Gate semantics and near-miss assembly** — Provisional — Four sub-decisions:
  - **(a) Location gate semantics** — *(superseded by AD-086)* — `candidate.location.remote_eligible` means "willing to work from a different location." Gate: `co_location_required=True` → pass if `candidate.location.city == scorecard.location.city` OR `candidate.location.remote_eligible is True`. **Refines AD-020's** "open-to-city" rule: the frozen model carries only a boolean, not per-city openness, so `remote_eligible` subsumes city-specific openness for MVP. `co_location_required=False` → any India location passes.
  - **(b) Near-miss ordering** — Cross-type: **availability misses rank above location misses** (availability near-misses are actionable — dates shift; location misses are structural). Within availability misses: smallest overshoot in days first. Within location misses: alphabetical by `candidate_email` (all location-miss candidates have `remote_eligible=False` by construction — G-LOC-2 passes anyone with `True` — so there is no meaningful gap metric to sort on).
  - **(c) Near-miss assembly** — The **orchestrator** (`dsm/cli/commands.py`) detects empty pool and builds `NoMatchResult.near_misses`. It recomputes gaps from the structured `Candidate` + `TargetProfileScorecard` objects it already holds — does NOT parse `Exclusion.detail`. `detail` stays human-readable only. Gates returns `(EligiblePool, ExclusionLog)` as typed. Rank returns `ShortlistResult` only — never `NoMatchResult`. Note: this supersedes `docs/structure.md` line 42 which previously assigned `NoMatchResult` to rank.
  - **(d) Top-3 near-misses** — `NoMatchResult.near_misses` is capped at 3 in the orchestrator. The model's list is unbounded; the cap is a presentation decision. Sufficient for "here's what came closest" without overwhelming the output.

## Ingestion architecture (`ee-ingestion-architecture.md`)

> Signed-off ingestion subsystem: raw sources → one query-ready canonical entity per consultant, plus the embedding written at ingest. Full detail in `ee-ingestion-architecture.md`.

- **AD-065 · Supply sources are CSV full snapshots** — Accepted — Supply is three CSV files (`beach.csv`, `rolling_off.csv`, `new_joiners.csv`), each a *full snapshot* carrying an `as of <date>` banner — **not xlsx**. The sheet a row appears in is the `AvailabilityState` discriminator; the banner date is parsed (never discarded) into `valid_as_of`. **Refines AD-012/AD-013.** Why: ee-ingestion-architecture §3.
- **AD-066 · Bronze/silver/gold layered, immutable, content-addressed store** — Accepted — Three physical layers: **bronze** (raw verbatim bytes + records), **silver** (typed, normalized, identity-resolved, per source), **gold** (one merged canonical entity per consultant). Each is immutable once written and addressed by content hash; **replay always runs from bronze**, never by re-reading the original source. Derivations are versioned and cached — the cache, not the model, is the source of truth for reproducibility. Why: determinism, cheap scoped re-merge/re-index, lineage. ee-ingestion-architecture §2/§4/§11.
- **AD-067 · `candidate_id = HMAC(email)` is the stable internal key** — Accepted — Email is the identity/dedup/join key but is tokenized to a stable `candidate_id` (HMAC) used as the internal key everywhere downstream (gold keyed by `candidate_id`; resumes/feedback join by the email they contain). **Name is never a join key** (colliding first names in the data). **Refines AD-012**: the raw email is the join input, but `candidate_id` is the in-pipeline identifier — `ingest` yields a map keyed by `candidate_id`. ee-ingestion-architecture §3.
- **AD-068 · Encrypted identity vault, retention-limited** — Accepted — Name + email are stripped from all derived/embedded text and held in an encrypted identity vault keyed by `candidate_id` (`name_vault_ref`/`email_vault_ref` point into it); retention-limited, gitignored, and supports purge-by-`candidate_id` erasure. **Supersedes AD-010's "mapping in-memory only" for ingestion-time identity**: the mapping is persisted but encrypted, and never reaches OpenRouter/Modal. The PseudonymisedLM in-memory mapping for live LLM calls is unchanged. ee-ingestion-architecture §4/§9.
- **AD-069 · Redact-first + NER-residual + outbound leak-scan** — Accepted — PII redaction is ordered: (1) deterministically strip known identifiers (name + email from the supply row), (2) Presidio NER for residual names + client orgs, (3) an outbound **leak-scan** that blocks any text still containing a known PII string and **fails the build/eval**. Clean-by-construction, defense in depth — does not rely on NER being perfect (important for Indian surnames and org names that read as ordinary words). **Refines AD-010/AD-011.** ee-ingestion-architecture §9/§10.
- **AD-070 · Snapshot reconciliation + tombstones + as-of freshness guard** — Accepted — Each run diffs the current `candidate_id` set against the prior set and **tombstones** consultants who have disappeared; the latest snapshot is authoritative for supply state. The banner date stamps `valid_as_of`; a **freshness guard** warns/refuses when a snapshot is stale relative to a role start. Why: current-state correctness, no stale ghost candidates. ee-ingestion-architecture §1/§5/§10.
- **AD-071 · Query-time reranking stage** — Accepted — A reranking stage sits between hybrid recall and the structured combine: a cross-encoder (recommended default **`BAAI/bge-reranker-base`**, hosted on Modal alongside the embedder per AD-074) or a single bounded LLM re-scores each role–candidate pair *jointly* for a precise top-k, which then feeds the structured combine. This is the precision lever the PRD's raw `0.6 × similarity` lacks; first-stage recall stays cheap and at current scale the eligible pool is reranked in full. **Does not change AD-030's structured score weights.** The default cross-encoder runs on Modal (no new OpenRouter step); an optional LLM rerank would be a *fourth* bounded OpenRouter call. ee-ingestion-architecture §8.
- **AD-072 · Capability-only embedding; hard skills matched via `skill_set`/BM25** — Accepted — The embedded passage is **capability-only and PII-free by construction**: include skills+proficiency, domains, projects, seniority evidence; exclude name/email/client-org/`candidate_id`, status/availability, grade-as-a-label, and **negations/absences** (embeddings cannot represent negation — absence is handled structurally by `skill_set` + `MergedSkill.demonstrated`). Dense vector → fuzzy adjacency only; a stated **hard skill is matched via `skill_set` (structured + BM25 sparse), never by cosine** (reinforces AD-033). Asymmetric passage/query formatting + a one-line contextual prefix per passage. ee-ingestion-architecture §8.
- **AD-073 · Evidence is verified quoted text** — Accepted — Every extracted fact carries an `EvidenceCitation` whose `quote` is a verbatim span **verified to exist in the source** before the extraction is accepted; a claim whose quote is not found is rejected. **Refines AD-040** with a concrete mechanism — no offsets to rot, no fabricated citations. ee-ingestion-architecture §6/§10.
- **AD-074 · BGE embedder hosted on Modal (serverless GPU)** — Accepted — The `bge-base-en-v1.5` embedder runs on **Modal** (serverless GPU) from MVP onward — **confirms AD-051**. The ingestion architecture's local-hosting implication (ee-ingestion-architecture §6 "locally hostable", §14) is **not adopted**: `modal/embedder.py` is the embed host and the `index/` embed-client calls it. Why: a single hosting path avoids a later local→Modal migration; Modal credits are ample at POC scale, and embedding text stays PII-free by construction (AD-011). ee-ingestion-architecture §6/§14.

## Silver stage & identity (a-002)

- **AD-075 · Relax `Location.city` to optional** — Accepted — `dsm/models.py::Location.city` becomes `str | None = None` (was required `str`). Why: the silver stage must represent `Remote (India)` consultants who have no base city (ee-ingestion §6/§15#3); an empty-string sentinel was dishonest typing. **Supersedes the `city: str` field shape in AD-060's frozen contract.** Consequence: gates already pass remote-eligible candidates via `remote_eligible` (AD-063a), and the `city == city` comparison degrades safely to `False` for `None`; near-miss rendering tolerates `None`. Backwards-readable — existing rows with a city are unaffected. Lanes B/C re-pull the contract.
- **AD-076 · `candidate_id` derivation + identity vault live in `dsm/pii/vault.py`; ingest→pii.vault import permitted** — Accepted — The `candidate_id = HMAC(email)` derivation (AD-067) and the encrypted identity vault (AD-068) live in **`dsm/pii/vault.py`** (under Lane C's `dsm/pii/` per AD-062). **Lane A seeds** a minimal `vault.py` this slice (deterministic `candidate_id` derivation + a `Vault` protocol + an in-memory store); **Lane C owns hardening the encrypted at-rest store later** — Lane C agrees to the placement and the Lane-A seeding. `NF-IMPORT-1` is **narrowed**: `dsm.ingest` may import `dsm.pii.vault` only; all other `dsm.pii` submodules (redact, leakscan, PseudonymisedLM) remain forbidden to ingest. Why: silver needs a stable internal key but cannot reach into the rest of `dsm.pii`; the vault is the legitimate identity store and the natural home for the tokenization. Consequence: a deliberate, signed-off narrowing of the PII import boundary (tech.md rule 1); the encrypted-at-rest impl + vault **write path** remain out of scope until the gold slice.

## Index build + upsert (a-005)

- **AD-081 · `skill_set` excludes feedback-denied skills** — Accepted — The index `skill_set` (the exact hard-skill filter + BM25 input) is built as `[s.name for s in gold.skills if s.demonstrated is not False]`: a feedback-refuted skill (`MergedSkill.demonstrated is False`, MG-5) is **excluded**, while confirmed (`True`) and unverified (`None`) skills are kept. The **same** predicate (`demonstrated is not False`) drives both the embedded passage's skill phrases (negation-free per AD-072) and `skill_set`, so the dense and structured views never disagree. Why: a stated hard skill is matched structurally, never by cosine (AD-033/AD-072); if a refuted skill stayed in `skill_set`, an exact/`ARRAY_CONTAINS` or BM25 match could credit a consultant for a skill feedback explicitly denied. Consequence: refuted skills are absent from retrieval entirely (dense + structured); they remain on the gold entity (with the conflict recorded) for explainability at serving time. See `specs/a-005-index-upsert/`.
- **AD-082 · No embed cache — `(gold_hash, model_version)` on the Milvus record gates re-embed** — Accepted — There is **no separate embed cache** (confirms AD-066's "the cache, not the model, is the source of truth" with the index record *being* that cache). Each `candidates` record stores both `gold_hash` (the gold content hash, GS-2) **and** `model_version` (= `config models.embedder`, the embedder id). On a run a candidate is re-embedded + upserted **only when its stored `(gold_hash, model_version)` pair does not BOTH match** the current pair; otherwise it is `skipped_unchanged`. Why: a data change flips `gold_hash`, but an embedder swap (e.g. `bge-base` → a larger model) leaves `gold_hash` unchanged while invalidating every vector — `model_version` is the term that catches it, so a model swap forces a full re-embed without a manual purge. Note: this index `model_version` is the **embedder** id and is deliberately distinct from `GoldCandidate.model_version` (the *reasoning* LLM used at enrich) — they gate different stages. Consequence: determinism/idempotency — identical gold + identical models → zero re-embeds and zero churn (NF-3). See `specs/a-005-index-upsert/`.
- **AD-083 · Add the `milvus-lite` embedded engine** — Accepted — `milvus-lite` is added to `pyproject.toml` dependencies (`uv lock` updated). `pymilvus` (already declared, AD-066/tech.md §11) is only the client SDK; it **cannot open a local `.db`** without the embedded `milvus-lite` engine. This is not a new technology choice — Milvus Lite is the already-sanctioned vector store (tech.md §11); this completes that choice so the store runs fully in-process with **no server and no network**. A probe confirmed it installs and runs offline on this platform (macOS, py3.12), including the BM25 `Function` + `SPARSE_FLOAT_VECTOR` + `enable_analyzer` path. Consequence: `make check` builds the real `candidates` collection in a tmp db with zero network (NF-1/NF-4). See `specs/a-005-index-upsert/`.
- **AD-084 · No outbound known-PII scan at index time — `embed_text` is PII-free by construction** — Accepted — **Supersedes the a-005 brief's "STILL run `assert_no_leak` on `embed_text`."** `build_embed_text`/`build_skill_set` read **only** capability fields of gold (`skills` with denied excluded, `domains[].value`, `projects`) and **never** touch `name_vault_ref`/`email_vault_ref`/`candidate_id` or any identity field, so identity cannot enter the passage bound for Modal — asserted by test (AC-6), not by a runtime scan. The dropped known-string scan would need the candidate's own name/email, which gold does not carry (vault refs only; `InMemoryVault` is non-persistent) — wiring it would couple `dsm index` to bronze + `DSM_CANDIDATE_ID_KEY` for only a narrow backstop (the de-anon-into-`projects` window, which enrich already never re-scans). Decision: rely on the construction guarantee; the indexer takes **no `known_pii`**, `dsm index` reads neither bronze nor the id key, and `dsm.index` does not import `dsm.pii` this slice. A **generic** outbound NER/org-dictionary scan (which also catches client-org names, without needing per-candidate identity) is **deferred to a later hardening phase**; `build_embed_text` is left as the single seam where it attaches. Why: defense-in-depth that buys nothing here is theatre — the honest guarantee is by-construction, and the generic scan is the real residual-PII hardening. Consequence: a deliberate, signed-off narrowing of the brief; tech.md rule 1 upheld (no PII reaches Modal — by construction here, by the generic scan later). See `specs/a-005-index-upsert/`.

## Ingestion fixes

- **AD-093 · Reconcile diffs against the full current supply roster, not just files re-landed this run** — Accepted — **Bug fix.** `dsm ingest` computed reconcile's *current* set as `{g.candidate_id for g in gold}`, but `gold` only contains candidates from files that were **LANDED** this run. Landing skips byte-identical files (`manifest.has_hash` → `SKIPPED`, [`land.py`](../dsm/ingest/land.py)) and the parse/silver loop processes **only LANDED** entries, so editing one supply file (e.g. removing a row from `Beach.csv`) left every candidate whose unchanged file was SKIPPED (all of `Rolling Off`, `New Joiners`) absent from the current set — and `reconcile(current, prior)` tombstoned them all. The `if not gold:` guard only covered the *all-skipped* (zero-candidate) re-run, not the *partial-change* case. **Decision:** since "the candidate universe is the supply sheets" (AD-013), reconcile's current set is the candidate_id roster across **all supply CSVs as they exist now** (LANDED *or* SKIPPED). The CLI rebuilds it by re-parsing the supply blobs (deterministic + LLM-free, so cheap) and deriving each row's candidate_id; the just-written gold ids are unioned in defensively. Enrichment/merge stay scoped to changed files (the LLM cost is why landing skips unchanged résumés/feedback — §11 enrichment cache still deferred); only the *membership* diff is widened. Strictly safer: the current set can only grow, so tombstones can only decrease. Known remaining edge (out of scope): emptying an *entire* supply file while others are SKIPPED yields zero gold this run and hits the `not gold` early-return, so that file's rows are not tombstoned until a run lands gold — rare; tracked for a later guard refactor. See [`dsm/cli/commands.py`](../dsm/cli/commands.py) `ingest`, regression test `tests/cli/test_ingest_cli.py::test_removing_one_supply_row_tombstones_only_that_candidate`.

- **AD-094 · Re-adding a departed supply row revives the candidate, even when landing dedups the file** — Accepted — **Bug fix, follow-on to AD-093.** After AD-093, removing a supply row tombstones exactly that candidate — but *re-adding* them did not reliably revive them. Two causes: (1) landing is content-addressed (`manifest.has_hash`), so reverting a file to byte-identical content seen on a prior run is **SKIPPED** — the row is never re-parsed/re-merged, so the normal "live merge overwrites the tombstone" path never runs; (2) with every supply file skipped, `gold` is empty and the old `if not gold:` early-return bailed before reconcile. Net: the candidate stayed tombstoned. **Decision:** (a) base the no-op guard on the **supply roster** (`if not current_ids:`) instead of `if not gold:` — the roster (AD-093) is the real "is this a snapshot?" signal; a pure idempotent re-run (empty gold, no departures, no revivals) still short-circuits with a "no changes" message. (b) Add an explicit **revive** pass: any candidate in the current supply roster whose on-disk gold is tombstoned but who was not re-merged this run (their file was SKIPPED) is flipped back to live via a new `reconcile.revive()` (symmetric with `tombstone()`; the tombstone preserved content, so it is a clean flip-back with a refreshed `gold_hash` → re-index). Candidates whose file *changed* (LANDED) are revived by the normal merge as before. Why a flip rather than re-merge: the tombstone already carried the full entity (incl. enrichment) forward, and the enrichment cache (§11) is deferred, so re-merging supply-only would *drop* prior resume/feedback. Known limitation (pre-existing, same root cause as AD-093's content-hash skip): if a supply file is reverted to an **older** version that differs from the one that last set the candidate's gold, the SKIP means gold can be stale — reverting to the *latest* state (the remove-then-re-add case) is correct; a general replay-from-bronze is deferred (§11). See [`dsm/cli/commands.py`](../dsm/cli/commands.py) `ingest`, [`dsm/ingest/reconcile.py`](../dsm/ingest/reconcile.py) `revive`, regression test `tests/cli/test_ingest_cli.py::test_readding_supply_row_revives_tombstoned_candidate`.

- **AD-095 · Eval harness architecture: code-based-first tiering, drop `promptfoo` PyPI placeholder** — Accepted — The eval harness uses a three-tier model: **Tier 1** (deterministic invariant evaluators — six pure Python functions over `ShortlistResult`/`NoMatchResult`, no LLM judge), **Tier 2** (signature regression — pin `clarify`/`score` DSPy output shape against cassette inputs), **Tier 3** (live smoke + cassette drift guard — real LLM, key-gated `skipif`). ## Near-miss enrichment

> Ratified as part of B-4 (`specs/b-004-near-miss-skill-verdict/`). AD-097 + AD-099 touch no frozen model;
> AD-098 + AD-100 are additive frozen-contract amendments (AD-060) — signed off at the B-4 `T-000-ADR` gate.
> (Renumbered from AD-095..098 → AD-097..100 when merging `main`, which had already taken AD-095/096 for c-002.)

- **AD-097 · Near-misses report their hard-skill verdict** — Accepted *(superseded by AD-099)* — A no-match near-miss for a candidate excluded *before* the hard-skill filter (`AVAILABILITY_MISMATCH`, `LOCATION_MISMATCH`) hid whether the candidate would clear the role's hard skills: the gates (`filter_candidates`) and `exact_hard_skill_filter` are two sequential narrowing stages, and a gate failure never reaches the filter (`run_match`), so "available 1 day after deadline" silently omitted a possible skill gap. **Decision:** every such near-miss states its hard-skill verdict in `gap_summary` — `"; clears all hard skills"` or `"; also missing N hard skill(s): …"` — computed by **reusing `exact_hard_skill_filter`** (deterministic, no adjacency per AD-033, proficiency floor per AD-072), so the no-match explanation can never drift from the real gate. Among `AVAILABILITY_MISMATCH` near-misses, those that also clear skills rank above those with a remaining gap (only the former are truly actionable — shift the date and they qualify), refining AD-063b ordering / the AD-088 actionability rationale. **No `NearMiss` field added** — the verdict rides in the existing `gap_summary` string; `Exclusion.detail` is still never parsed (AD-063c). Deterministic + LLM-free (Golden rule 2 / AD-002). Rejected alternative: a typed `clears_hard_skills: bool` field — cleaner for evals but a frozen-contract amendment for marginal gain; deferred. See [`dsm/cli/commands.py`](../dsm/cli/commands.py) `build_near_misses`, `specs/b-004-near-miss-skill-verdict/`.

- **AD-098 · `NearMiss.selection_rationale` (LLM, top-3 only, via `PseudonymisedLM`)** — Accepted — Add an optional `selection_rationale: str | None = None` field to the frozen `dsm/models.py::NearMiss`, carrying a short LLM-written "why consider this candidate once the gap is resolved" — the upside that complements `gap_summary`'s gap. **Frozen-contract amendment (AD-060)** — additive, optional, defaulted, so every existing `NearMiss(...)` construction keeps validating. **Decision:** generate the rationale **only for the ≤3 near-misses actually shown** (after ordering + the AD-063d cap), through an injected `NearMissRationalePredictor` seam built over `PseudonymisedLM` and fed only PII-free structured inputs (skills, feedback, gap) — no `name`/`email` to any provider (Golden rule 3), mirroring `make_score_predictor`. The seam threads `run_match` → `_no_match`; tests inject a fake (no network, NF-1), the live predictor is built at the CLI edge. On predictor error the field stays `None` and the near-miss is still shown (parallel to `score_candidate`'s skip-on-error). The rationale is **explanation only** — it never changes the near-miss set or ordering, so eligibility stays deterministic + LLM-free (AD-002) and the determinism invariant continues to bind the set + ordering + `gap_summary`, not the prose (same scoping as the shortlist `CandidateAssessment.narrative`). Parallels AD-092 (a prior additive frozen amendment ratified mid-slice). Rejected alternative: a deterministic LLM-free strengths summary — simpler and bit-deterministic, but richer LLM prose was chosen at the design fork. See [`dsm/models.py`](../dsm/models.py) `NearMiss`, [`dsm/cli/commands.py`](../dsm/cli/commands.py), [`dsm/match/score.py`](../dsm/match/score.py), `specs/b-004-near-miss-skill-verdict/`.

- **AD-099 · A near-miss must clear the hard skills (skill-failers are not near-misses)** — Accepted — Refines what qualifies as a near-miss: a near-miss is a candidate **one fixable decision** from eligibility — it fails only a *negotiable* gate (availability or location, which a human can shift/relax) **and** clears the *non-negotiable* hard skills (exact match, no adjacency — AD-033/072). **Decision:** `build_near_misses` keeps only `AVAILABILITY_MISMATCH`/`LOCATION_MISMATCH` candidates that clear `exact_hard_skill_filter`; everyone else — pure `HARD_SKILL_MISMATCH` misses **and** gate misses that *also* fail a hard skill — is dropped from near-misses. They are **not** lost: the full `ExclusionLog` (and `dsm explain`) still records every excluded candidate with their reason — near-misses are the curated "closest *actionable*" view, the exclusion log is the transparency layer. **Supersedes AD-088** (which surfaced hard-skill misses as near-misses) and **AD-097** (which instead *labelled* the skill gap in `gap_summary` + sub-ordered availability misses by skill clearance). Consequence: `gap_summary` reverts to the plain negotiable-gap wording (`"available N days after deadline"`, `"in X, not in onsite set for Y"`) since every near-miss now clears skills by construction; the AD-097 skill-verdict suffix and the sub-rank are removed; ordering returns to AD-063b (availability before location, by overshoot then email). Known edge: a no-match where *every* candidate lacks the hard skill yields **zero** near-misses — correct (none are actionable); the `reason` string + exclusion log still explain it. AD-098 (`selection_rationale`) is unaffected — it still annotates whatever near-misses remain. Why: "available 1 day late" for someone who also lacks a hard skill is a dead end dressed as a near-miss; hard skills are a gate in all but name (`exact_hard_skill_filter` removes them from the pool), so failing one is disqualifying, not "near". Rejected alternatives: (a) keep current AD-097 labelling — surfaces non-actionable rows; (b) drop only the double-misses but keep pure skill-misses — still presents an unactionable (non-negotiable) gap as a near-miss. No frozen-model change. See [`dsm/cli/commands.py`](../dsm/cli/commands.py) `build_near_misses`, `specs/b-004-near-miss-skill-verdict/`.

- **AD-100 · `NoMatchResult.closest_on_skills` — the skill-axis counterpart to near-misses** — Accepted — Add an optional `closest_on_skills: list[NearMiss] = Field(default_factory=list)` to the frozen `dsm/models.py::NoMatchResult`. **Frozen-contract amendment (AD-060)** — additive, optional, defaulted, so existing constructions stay valid. AD-099 made a near-miss "one *negotiable* decision away" (clears hard skills, fails only availability/location); that correctly dropped skill-failers, but a no-match where everyone available is a skill short (e.g. ROLE-05: Remote-India, hard skills java/spring boot/postgresql) then shows **zero** near-misses with no view of who came closest on skills. **Decision:** add a **separate, disjoint** list of candidates who cleared **both** gates and failed **only** the hard-skill filter (reason `HARD_SKILL_MISMATCH`), ranked by fewest hard-skill gaps (missing + below-floor count) then `candidate_id`, capped at 3 (AD-063d), each optionally annotated with the AD-098 `selection_rationale` (reused as-is — its prompt already covers a skill gap). It is the structural mirror of a near-miss: near-miss = clears skills + one negotiable gap; closest-on-skills = clears both gates + a skill short. The two lists never overlap (each candidate has exactly one exclusion reason). The gap is computed by the **same** logic as `exact_hard_skill_filter` via a shared `hard_skill_gap` helper (one source of truth, AD-072/033, no adjacency); `Exclusion.detail` is never parsed (AD-063c). Reuses the `NearMiss` type — no new model. Empty when the no-match is gate-only or empty-input (no `HARD_SKILL_MISMATCH` exclusions). Rejected alternative: re-admitting skill-failers into `near_misses` — exactly what AD-099 removed; a separate field keeps the actionable list clean. See [`dsm/models.py`](../dsm/models.py) `NoMatchResult`, [`dsm/cli/commands.py`](../dsm/cli/commands.py) `build_closest_on_skills`, [`dsm/index/retrieve.py`](../dsm/index/retrieve.py) `hard_skill_gap`, `specs/b-004-near-miss-skill-verdict/`.

- **AD-101 · Real PseudonymisedLM query-time boundary (call-context known-PII + redact/leak-scan/de-anonymise)** — Accepted — `pii/PseudonymisedLM` stops being a no-op pass-through and becomes the real outbound boundary for **query-time** LLM calls (it already was, manually, inside ingest `enrich`). **The leak it closes:** the serving `Candidate` is hydrated from **gold**, whose `feedback`/`projects` free-text is **de-anonymised** (enrich restores real names before writing gold, AD-077/078); `score` ([`dsm/match/score.py`](../dsm/match/score.py)) sends `candidate_feedback` + `profile_summary` to the LLM through `PseudonymisedLM`, so with the stub a name embedded in feedback/projects reached OpenRouter. **Mechanism (Approach A, chosen over NER-only and over re-tokenising gold):** the wrapper only sees a formatted prompt, so the candidate's known identifiers are supplied via a **`contextvars.ContextVar` set by a `pii_context(known_pii)` context manager**; `__call__` then redacts both `prompt` and `messages` (DSPy 3.x) under **one stable `Redactor` mapping** (deterministic known-identifier strip first — the AD-069 load-bearing guarantee — then NER residual), runs `assert_no_leak` on every outbound fragment as a **hard gate** (`PIILeakError` blocks the call), forwards, and **de-anonymises the response** so echoed placeholders (verbatim citation quotes) are restored. **Unset context = full pass-through** — `clarify` carries role text only, no candidate PII (`ee-query-architecture.md` §7), so it is invoked without `pii_context` and behaves exactly as before; a context set to `[]` still engages NER. **Wiring lives at the CLI composition root** (which already owns `PseudonymisedLM` + `GoldCandidateStore`): the injected `ScorePredictor` is wrapped to resolve the candidate's identity from the vault (by `candidate_id`, AD-102) and enter `pii_context` per candidate — so **`dsm.match` stays free of any `dsm.pii` import** and a new import contract enforces it. Why: satisfies AD-010/AD-069 at query time without weakening the deterministic guarantee or threading PII through the pure pipeline. Scope held: this slice does **not** de-pseudonymise names in the final shortlist output (output stays `candidate_id`). See `specs/c-003-pseudonymised-lm-boundary/`, `ee-query-architecture.md` §6.0/§6.8.

- **AD-102 · Minimal persistent `FileVault` + `get_identity` read path; encryption deferred** — Accepted — Approach A (AD-101) needs the candidate's real `name`/`email` at **query time** to drive the deterministic redact-first pass and the leak-scan, but nothing persists identity across the `ingest`→`match` **process boundary** today: gold carries only `name_vault_ref`/`email_vault_ref` pointers (AD-068), `known_pii[cid] = [name, email]` is built from bronze during ingest and discarded in-process, and `InMemoryVault` is non-persistent (AD-076/AD-084). **Decision:** add a `FileVault` — a **file-backed, gitignored** identity store keyed by `candidate_id` (default `data/identity/vault.json`, written atomically) — and add **`get_identity(candidate_id) -> tuple[str, str] | None`** to the `Vault` Protocol (missing id → `None` → empty known-PII list → NER-only, never a crash). **Additive frozen-contract amendment (AD-060)** — the protocol gains a method; existing implementations add it. `dsm ingest` **writes** it from the `known_pii` it already computes ([`dsm/cli/commands.py`](../dsm/cli/commands.py)); the query plane **reads** it (at the CLI, per AD-101) — the only new consumer of raw identity at query time. **Encryption at rest, retention limits, and purge-by-id remain explicitly deferred to the AD-068 hardening slice** (the store **MAY be plaintext** now; the module docstring + `docs/tech.md` §PII record this so it isn't re-litigated). Gold is unchanged (refs only); no raw identity is logged or committed. Why: this is the minimum that makes the deterministic guarantee honest at query time; it stores at-rest only what already lived in-process during ingest, and does not touch the *outbound* guarantee (redact-first + leak-scan). See `specs/c-003-pseudonymised-lm-boundary/`.

## PII boundary posture (post-c-003)

- **AD-103 · Vault-miss at query time is warn-only, not fail-closed** — Accepted — Follow-on to AD-101/102. When `vault.get_identity(candidate_id)` returns `None` at score time, the known-PII list is empty, so the deterministic strip + `assert_no_leak` have nothing to match on; protection falls back to NER, which itself degrades to a no-op when `en_core_web_lg` is absent. **Decision: keep the current warn-only behaviour** — log `score.vault_miss_reduced_redaction` (PII-safe: `candidate_id` only) and proceed — rather than fail-closed (skip/refuse the candidate). Why: a vault path mismatch or stale/un-built vault would, under fail-closed, silently drop **every** candidate — turning an operational/config error into a confusing total outage, which is worse UX for no real safety gain at POC scale (the data is synthetic; the deterministic gate already blocks *known* leaks when the vault is populated). The residual exposure (empty vault **and** degraded NER → a candidate's de-anonymised gold free-text reaches the provider) is **accepted for now** and tracked: it is subsumed by the AD-084 generic outbound NER/org scan (the real residual-PII hardening) and the AD-068 vault hardening. A startup check that errors when `pii.ner_enabled=true` but the spaCy model is missing was **considered and declined** for this round (revisit with AD-084). Rejected alternative: fail-closed — surfaced as a deliberate posture call so it isn't silently re-litigated. No code change (status quo ratified). See [`dsm/cli/commands.py`](../dsm/cli/commands.py) `_pii_aware_score_predictor`, `docs/progress.C.md`.

## AI eval layer (c-004)

> Ratified as part of C-4 (`specs/c-004-ai-eval-layer/`). No frozen-contract amendments.
> These decisions add the model-quality evaluation tier that the c-002 deterministic
> invariants cannot cover: faithfulness judging, retrieval quality metrics, and the
> labelled golden set they depend on. All **non-gating** — `make eval` only.

- **AD-104 · Hand-labelled golden set for AI-quality evals** — Accepted — Create a hand-labelled golden set (`tests/fixtures/golden_set.json`, 20–40 cases) binding seed roles to expected shortlists, expected relevant sets (for Recall@K), and per-candidate faithfulness labels (y/n). The file carries `_meta.review_status: "draft"|"signed_off"` — all eval tests that depend on label correctness (judge validation, metric reporting) gate on `review_status == "signed_off"` and skip otherwise. **Labels are drafted by machine** from fixture definitions + pipeline output analysis (deterministic invariants already tell us who passes gates, who is excluded, and what the ordered shortlist looks like under the cassette LM). **They are not trusted until a human reviews and signs off** — generated labels carry the risk of encoding pipeline bugs as ground truth. Why: the narrative-faithfulness judge (AD-095 deferred) needs labelled validation data with TPR/TNR ≥ 80% to be adopted; Recall@K needs a relevant set; ranking metrics need expected orderings. Without labels these metrics are either unvalidatable or unmeasurable. See `specs/c-004-ai-eval-layer/`.

- **AD-105 · DeepEval G-Eval faithfulness judge (validated, non-gating)** — Accepted — Wire a `deepeval.metrics.GEval` faithfulness judge that scores whether a candidate narrative follows from the cited evidence + candidate data. The judge is **validated against the golden set** — adopted only if TPR ≥ 0.80 and TNR ≥ 0.80 against signed-off human labels; if it fails validation, the test skips (not fails). The judge **SHALL NOT** evaluate objective properties already covered by the six deterministic invariants (gates, PII, citations, adjacency, hard-skill-exclusion, determinism) — the LLM judge covers only the subjective residual: fabrication, contradiction, and consistency between narrative and sub-scores. Runs under `make eval` only (`eval_live`, key-gated `skipif`). **Never in `make check`** (non-deterministic + costs keys). The six deterministic invariants remain the commit gate. Why: an unvalidated LLM judge is worse than no judge — it adds cost and false confidence. Validation-first (TPR/TNR ≥ 80%) ensures the judge earns its place before results are reported. AD-095 deferred this until labelled data existed; AD-104 provides it. See `specs/c-004-ai-eval-layer/`.

- **AD-106 · Retrieval Recall@K + contextual precision (non-gating)** — Accepted — Compute deterministic Recall@K and contextual precision over the golden set's `expected_relevant_set`. Recall@K = |retrieved ∩ relevant| / |relevant| where K = `index.rerank.top_k`. Contextual precision = |relevant ∩ top-K| / K. Computed over the post-rerank set (the candidates that enter scoring). Additionally, run `deepeval.metrics.ContextualRecallMetric` / `ContextualPrecisionMetric` for the LLM-judged retrieval view (signed-off labels only). Skipped when `index.recall.enabled = false` (trivially 1.0). Runs under `make eval` only. Why: hybrid recall is ON (AD-089 default `true`), so retrieval quality is now a meaningful axis; without metrics on it, retrieval regressions are invisible. See `specs/c-004-ai-eval-layer/`.

- **AD-107 · Final human-facing identity de-anonymisation at the CLI output edge** — Accepted — The query pipeline (`dsm.match`) keeps emitting **pseudonymised** results (identity field = `candidate_id`, AD-091); a new render step `render_identities(result, vault)` in [`dsm/cli/commands.py`](../dsm/cli/commands.py) (the composition root that already owns the `FileVault`) substitutes the real `(name, email)` from the vault into **every** identity-bearing output field — `ranked_assessments[].candidate.email`/`.name`, both `NearMiss` lists' `candidate_email`/`name`, and every `Exclusion.candidate_email` (decision: de-anon *everything incl. the exclusion log*). Rejected: a formatted/tabular terminal view (deferred — JSON-with-identity chosen); de-anonymising inside `dsm.match` (would break the import boundary). See `specs/b-003-human-readable-shortlist/`.

## Runtime / environment

- **AD-108 · Set `KMP_DUPLICATE_LIB_OK=TRUE` at the CLI entrypoint (macOS duplicate-OpenMP workaround)** — Accepted — `dsm/cli/main.py` sets `os.environ.setdefault("KMP_DUPLICATE_LIB_OK", "TRUE")` as its **first statement**, before any import. **The crash it fixes:** three pip wheels each statically bundle a *different* `libomp.dylib` — `faiss-cpu` (via `milvus-lite`, the vector index), plus `torch` and `scikit-learn` (via the spaCy/Presidio NER stack the PII boundary uses). Any `dsm match` that produces a **shortlist** loads both the index (faiss) **and**, at the scoring stage, the NER stack (torch) into one process; the second OpenMP runtime to initialise calls `abort()` (`OMP: Error #15`, exit 134). No-match roles short-circuit before scoring, so they never co-load and never crashed — which is why the failure looked role-dependent (it tracks shortlist-vs-no-match, not role id). **Why this approach:** the var must be set before the OpenMP-linked libs load (OpenMP reads it at library-init), so the entrypoint — ahead of its own imports — is the correct, reliable place; being committed, it covers every subcommand (`match`/`explain`/`ingest`/`index`) and every machine + CI, unlike a per-developer shell `export`. `setdefault` lets an explicit `KMP_DUPLICATE_LIB_OK=FALSE` still win (to surface conflicts while debugging). **Risk accepted:** OpenMP warns this *can* "silently produce incorrect results", but that failure mode requires colliding nested parallel regions; this workload is single-pass CPU inference over synthetic POC data, so the practical risk is negligible. The only true fix (one centralised `libomp`) means moving the toolchain to conda-forge — rejected as far too heavy for a POC and contrary to the committed `uv`/pip choice (`docs/tech.md`). Rejected alternatives: per-developer shell `export` (not shared/reproducible — fails teammates + CI); a project `.env` (`uv run` doesn't auto-load it); dropping faiss or the NER stack (each is load-bearing — the index resp. the PII boundary). Lives at the composition root, so `dsm.match`/`dsm.pii` are untouched. See `docs/tech.md` § Stack, `dsm/cli/main.py`.
